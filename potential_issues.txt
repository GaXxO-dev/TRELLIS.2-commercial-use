ISSUE 2: DEPTH VALUE CONVENTION
================================================================================

STATUS: [VERIFIED] - NOT A BUG, JUST DIFFERENT CONVENTION

SEVERITY: INFO - Not causing rendering errors

DESCRIPTION:
  Depth values in rasterizer output differ by convention:
  - Fork (DRTK):    depth ∈ [9.66, 10.27] (camera-space z)
  - Official:       depth ∈ [0.81, 0.82] (normalized depth z/w in NDC)

EXPLANATION:
  nvdiffrast outputs z/w (normalized device coordinates depth).
  DRTK outputs z_camera (actual distance from camera).
  
  Both are CORRECT for their respective conventions. The difference
  does NOT cause rendering errors - depth ordering is preserved.

LOCATION:
  - trellis2/utils/drtk_compat.py line ~203 in rasterize()
    rast[0, ..., 2] = depth[0]  # camera-space depth

NOTE: This is INTENTIONAL. z_camera is actually more physically
      meaningful than z/w for many use cases. DO NOT "FIX" THIS.

ISSUE 4: BARYCENTRIC COORDINATE MAPPING
================================================================================

STATUS: [VERIFIED] - Mapping is correct

SEVERITY: MEDIUM - Could cause interpolation errors

DESCRIPTION:
  DRTK and nvdiffrast use different barycentric coordinate conventions:
  
  DRTK:     bary_img has shape [1, 3, H, W] with weights (w0, w1, w2)
            for vertices v0, v1, v2 respectively
  
  nvdiffrast: rast[..., 0:2] has (u, v) where:
            u = weight for vertex 1
            v = weight for vertex 2
            w0 = 1 - u - v (weight for vertex 0)

  The current mapping in drtk_compat.py:
    rast[0, ..., 0] = bary[0, 1]  # u = weight for vertex 1
    rast[0, ..., 1] = bary[0, 2]  # v = weight for vertex 2

LOCATION:
  - trellis2/utils/drtk_compat.py lines ~180-190 in DRTKContext.rasterize()

VERIFICATION RESULTS:
  - Barycentric u mean: 0.3339 (fork) vs 0.3292 (official) - reasonable match
  - Barycentric v mean: 0.3323 (fork) vs 0.3368 (official) - reasonable match
  - Values in valid range [0, 1] for geometry pixels
  - Interpolated normals have reasonable range [-1, 1]
  - Interpolated depth values correct (camera-space z)
  - Visual check: normal_00.png shows proper surface normals

DESCRIPTION:
   The voxel attributes (mesh.attrs) from the texture decoder differed between
   fork and official even though:
   - Mesh vertices and faces are IDENTICAL
   - Same random seed (42)
   - Sparse structure coords are IDENTICAL
   
   Comparison (base_color, metallic, roughness, alpha channels):
   
   | Channel      | Fork Mean | Official Mean |
   |--------------|-----------|---------------|
   | base_color   | 0.2047    | 0.0988        |
   | metallic     | 0.7126    | 0.8960        |

   The fork's sample_tex_slat() used tex_slat_normalization for denormalization

   copy-paste bug in official, the fork must match official's behavior to
   produce consistent renders.

FIX:
   Changed sample_tex_slat() to use shape_slat_normalization instead of
   tex_slat_normalization, matching official's (incorrect but consistent)
   behavior.

LOCATION:
   - trellis2/pipelines/trellis2_image_to_3d.py sample_tex_slat()

VERIFICATION:
   - attrs mean values now match official exactly
   - base_color mean: 0.0988 (matches official)
   - metallic mean: 0.8960 (matches official)
   - roughness mean: 0.2921 (matches official)
   - alpha mean: 0.7248 (matches official)

ISSUE 12: PBR MATERIALS POOR QUALITY (CRITICAL - NEEDS FIX)
================================================================================

STATUS: [IN_PROGRESS] - grid_sample_3d sparse voxel sampling issue

SEVERITY: CRITICAL - Material rendering broken, must match official exactly

DESCRIPTION:
     The T model is BARELY VISIBLE in PBR material outputs despite correct
     mask coverage (67266 pixels). THIS IS UNACCEPTABLE - materials must be
     nearly identical to official nvdiffrast implementation.

     Key measurements:
     - mask_pbr_00.png: Coverage correct (67266 vs 67280 official) ✓
     - normal_pbr_00.png: Normal values correct ✓
     - base_color_00.png: Only 2898/201798 nonzero pixels vs 157150/201840 official ✗
     - Materials ~60x dimmer than official ✗

INVESTIGATION COMPLETED:
     The PBREnvironmentLight.shade() function works CORRECTLY - tested with
     identical inputs produces identical outputs:
     - Diffuse convolution matches official
     - BSDF LUT matches official
     - shade() function output matches official

     The issue is NOT in PBREnvironmentLight. The issue is in grid_sample_3d:

     EVIDENCE:
     1. grid_sample_3d works correctly when tested directly with sparse voxel coords
        (querying at voxel centers returns correct values)
     
     2. When given official's xyz_voxel values, fork's grid_sample_3d produces
        CORRECT output (verified in Issue 9)
     
     3. Fork's xyz_voxel values differ from official's because DRTK's rasterizer
        produces different barycentric coordinates at edges

     4. The sparse voxel grid has ~1.47M voxels, but the mesh surface only
        passes through SOME of them. When grid_sample_3d queries at a position
        where the surrounding 8-voxel trilinear cell has MISSING corners, those
        corners contribute 0 to the interpolation.

ROOT CAUSE ANALYSIS:
     DRTK's rasterize() produces different triangle IDs at edge pixels than
     nvdiffrast. This leads to different barycentric weights, which leads to
     different interpolated vertex positions (xyz). When these xyz positions
     are converted to voxel coordinates and passed to grid_sample_3d, they
     land at positions where the sparse grid has fewer filled 8-corner cells.
     
     The flex_gemm grid_sample_3d trilinear interpolation returns (weighted sum)
     where missing corners contribute zero, making the output very dim.

COMPARISON WITH Issue 9:
     - Issue 9 claimed "RESOLVED - NOT A BUG" based on the logic that different
       rasterizers produce different results
     - HOWEVER: The user requires PBR materials to be IDENTICAL to official
     - The ~60x brightness difference is NOT acceptable
     - This MUST be fixed even if it requires adjusting how xyz is computed

NEXT STEPS TO FIX:
     Option A: Fix xyz interpolation to match nvdiffrast more closely
       - Investigate if there's a Y-axis or other coordinate difference
       - Check if barycentric coordinate convention is different
     
     Option B: Implement sparse-aware grid_sample_3d
       - Count available corners in trilinear cell
       - Normalize by number of available corners
       - This is what nvdiffrast likely does internally
     
     Option C: Pre-compute dense voxel grid from sparse representation
       - Convert sparse voxel grid to dense grid before sampling
       - More memory but guarantees all 8 corners exist

LOCATION:
     - trellis2/renderers/pbr_mesh_renderer.py lines 351-358 (grid_sample_3d call)
     - trellis2/utils/drtk_compat.py (interpolate function)
     - flex_gemm/ops/grid_sample/grid_sample.py (grid_sample_3d implementation)

DEBUG FILES SAVED:
     - test_output_pbr/debug/*.npy - Fork's debug outputs
     - test_output_pbr_official/debug/*.npy - Official's debug outputs
     - compare_pbr.py - Comparison script
